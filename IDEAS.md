## Большая задача: переработка логики анализа

### Проблемы текущей механики

1. **Score-формула** — `100 - penalty + bonus` может дать >100. Бонусы за «нет циклов» завязаны на подстроку в `issue.title` — хрупко.
2. **Не все параметры учитываются** — `frequency`, `synchronous` на связях используются частично. `requestRate`, `hitRate`, `reliability` почти не влияют.
3. **Нет весов правил** — все issues одного severity штрафуются одинаково.
4. ~~**Метрики без объяснений**~~ — есть METRIC_TOOLTIPS.
5. **Семантика связей** (TASK-071) — правила не проверяют осмысленность: reads/writes для БД, calls для сервисов, subscribes/emits для event-driven.

### Предложения

- [ ] **Документировать формулы** — явно описать расчёт score, grade, каждой метрики.
- [ ] **Вынести веса правил** — конфиг `ruleWeights: { S02: 20, L03: 15, ... }`.
- [ ] **Ограничить score 0–100** — `Math.max(0, Math.min(100, raw))`.
- [x] **Tooltip к метрикам** — METRIC_TOOLTIPS в analysis-metrics.
- [ ] **Unit-тесты для правил** — фикстуры-графы, ожидаемые issues.
- [ ] **Правила семантики связей** — «reads/writes только для БД/кэша», «calls только для API/сервисов» → issues при нарушении.

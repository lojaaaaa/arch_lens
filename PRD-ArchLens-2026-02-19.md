## 1. Обзор продукта и цели

### 1.1 Краткое описание

**ArchLens** — веб-сервис, позволяющий пользователю визуально проектировать архитектуру программной системы в формате графической нотации (System Design), а затем получать автоматический анализ спроектированной архитектуры с оценкой качества, выявлением ошибок, рекомендациями по улучшению и (в будущем) AI-генерируемыми советами.

### 1.2 Проблема

- Студенты и начинающие разработчики проектируют архитектуру «вслепую», без обратной связи о качестве решений.
- Подготовка к секциям System Design на собеседованиях требует практики, но нет инструмента, который бы давал объективную оценку спроектированной системы.
- Существующие инструменты (Excalidraw, draw.io, Miro) позволяют рисовать схемы, но не анализируют их содержание — это просто canvas без семантики.

### 1.3 Решение

ArchLens объединяет визуальный редактор архитектуры с семантическим анализатором. Каждый элемент схемы — это не просто прямоугольник, а типизированная сущность с параметрами, которые влияют на анализ. Связи между элементами тоже типизированы. Это позволяет:

- Строить граф зависимостей архитектуры
- Применять rule-based анализ на основе архитектурных паттернов
- Оценивать нагрузочные характеристики
- Давать конкретные рекомендации по улучшению

### 1.4 Ключевые метрики успеха

| Метрика | Цель |
|---------|------|
| Количество правил анализа | ≥ 20 для MVP |
| Точность выявления проблем | ≥ 80% на типовых архитектурах |
| Время от открытия до первого анализа | < 5 минут (UX) |
| Оценка удобства интерфейса | ≥ 4/5 (юзабилити-тестирование) |

## 3. Основные функции и функциональность

### 3.1 Визуальный редактор архитектуры

#### 3.1.1 Слои и типы узлов

Архитектура строится в рамках трёх слоёв. Каждый слой содержит типы узлов (нод):

**UI / Frontend:**

| Тип узла | Описание | Параметры |
|----------|----------|-----------|
| `Page` | Страница приложения | Маршрут, кол-во компонентов, использование состояния (нет / локальное / глобальное), частота обновления |
| `UI Component` | Визуальный компонент | Тип (ввод / таблица / кнопка / кастомный), вложенные компоненты, кол-во props, тип состояния (нет / локальное / контекст / глобальное), частота рендера |
| `State Store` | Хранилище состояния | Технология (Redux / Zustand / Context / localStorage / sessionStorage), кол-во подписчиков, частота обновления |

**Backend:**

| Тип узла | Описание | Параметры |
|----------|----------|-----------|
| `API Gateway` | Точка входа API | Кол-во эндпоинтов, частота запросов, требуется ли авторизация |
| `Service` | Бизнес-сервис | Кол-во операций, внешние вызовы, наличие состояния (stateful/stateless) |

**Data:**

| Тип узла | Описание | Параметры |
|----------|----------|-----------|
| `Database` | База данных | Тип (SQL / NoSQL), кол-во таблиц/коллекций, соотношение чтение/запись |
| `Cache` | Кэш | Тип (Redis / in-memory), hit rate |
| `External System` | Внешний API/сервис | Тип (auth / payment / analytics / storage / notification / другое), протокол (REST / GraphQL / SOAP / gRPC), надёжность (SLA), задержка (мс), rate limit |

#### 3.1.2 Общие параметры каждого узла

| Параметр | Описание | UX-представление |
|----------|----------|------------------|
| `complexity` | Сложность компонента | Ползунок 1–5 с подписями: «Простой → Сложный» |
| `criticality` | Критичность для системы | Ползунок 0–3 с подписями: «Некритичный → Ключевой» |

> **UX-принцип:** Все числовые параметры должны иметь человеко-понятные подписи, подсказки (tooltips) с примерами и визуальные индикаторы (цвет/иконки). Пользователь не должен гадать, что значит «requestRate = 50» — интерфейс должен показать «~50 запросов/сек — высокая нагрузка».

#### 3.1.3 Типы связей (рёбер)

| Тип связи | Описание | Визуальный стиль |
|-----------|----------|------------------|
| `calls` | Вызывает (синхронный API-вызов) | Сплошная линия со стрелкой |
| `reads` | Читает данные | Пунктирная линия со стрелкой |
| `writes` | Записывает данные | Жирная линия со стрелкой |
| `subscribes` | Подписывается на события | Волнистая/точечная линия |
| `depends_on` | Зависит от | Штриховая линия |
| `emits` | Отправляет события | Штрих-пунктирная линия |

**Параметры связи:**
- `frequency` — частота взаимодействия (ползунок: редко → постоянно)
- `synchronous` — синхронное/асинхронное взаимодействие (toggle)

#### 3.1.4 Операции редактора

| Операция | Описание | Acceptance Criteria |
|----------|----------|---------------------|
| Добавление узла | Через сайдбар или drag-and-drop | Узел появляется на canvas с дефолтными параметрами; тип и слой определены автоматически |
| Удаление узла | Выбор + Delete или контекстное меню | Узел и все связанные рёбра удаляются; если узел участвует в анализе — предупреждение |
| Создание связи | Drag от выхода узла к входу другого | Связь создаётся с типом по умолчанию `calls`; тип можно изменить в properties panel |
| Редактирование параметров | Клик/двойной клик по узлу → properties panel | Все параметры узла доступны в боковой панели с валидацией |
| Перемещение узлов | Drag на canvas | Плавное перемещение с привязкой к сетке (snap-to-grid опционально) |
| Zoom/Pan | Scroll + drag по пустому пространству | Плавный zoom с min/max ограничениями; minimap для навигации |

#### 3.1.5 UX-требования к редактору

- **Onboarding:** При первом открытии — пустой canvas с подсказкой «Перетащите элемент из панели слева, чтобы начать проектирование»
- **Быстрые действия:** Горячие клавиши (⌘S сохранить, ⌘O импорт, Delete удалить выбранное)
- **Визуальное разделение слоёв:** Ноды разных слоёв визуально различимы (цвет/форма/иконка): Frontend — синий, Backend — зелёный, Data — оранжевый
- **Auto-layout (future):** Автоматическое выравнивание нод по слоям
- **Undo/Redo (future):** История действий

### 3.2 Система анализа

#### 3.2.1 Архитектура анализатора

Анализ состоит из четырёх слоёв, подключаемых последовательно:

```
┌─────────────────────────────────────────────┐
│  Layer 1: Structural Analysis               │ ← MVP
│  (граф, связи, orphans, cycles, bottleneck) │
├─────────────────────────────────────────────┤
│  Layer 2: Pattern Analysis                  │ ← MVP
│  (антипаттерны, best practices, паттерны)   │
├─────────────────────────────────────────────┤
│  Layer 3: Load Estimation                   │ ← MVP
│  (нагрузка, bottleneck по метрикам)         │
├─────────────────────────────────────────────┤
│  Layer 4: AI Recommendations                │ ← Future (DeepSeek)
│  (текстовые рекомендации от LLM)            │
└─────────────────────────────────────────────┘
```

Каждый слой — отдельный модуль, реализующий интерфейс `AnalysisRule`. Это позволяет:
- Добавлять правила независимо
- Тестировать каждое правило изолированно
- В будущем подключить AI-слой без изменения остальных

#### 3.2.2 Правила анализа — Layer 1: Structural

| ID | Правило | Severity | Описание |
|----|---------|----------|----------|
| S01 | Orphan nodes | info | Узлы без связей — возможно, забыты |
| S02 | Cyclic dependencies | critical | Циклы в графе зависимостей |
| S03 | High fan-out | warning/critical | Узел с > N исходящих связей (bottleneck) |
| S04 | High fan-in | warning | Узел с > N входящих связей (single point of failure) |
| S05 | Missing data layer | warning | Backend-сервис не связан ни с одной БД/кэшем |
| S06 | Frontend → Database direct | critical | Фронтенд напрямую обращается к БД (минуя backend) |
| S07 | No API Gateway | warning | Backend-сервисы доступны без gateway |
| S08 | Disconnected layers | warning | Один из слоёв полностью изолирован от остальных |
| S09 | Single point of failure | critical | Критический узел без резервирования (criticality ≥ 2, fan-in ≥ 3) |
| S10 | Redundant edges | info | Дублирующие связи между одной парой узлов |

#### 3.2.3 Правила анализа — Layer 2: Pattern

| ID | Правило | Severity | Описание |
|----|---------|----------|----------|
| P01 | God Service | critical | Сервис с > 10 операциями и > 5 внешних вызовов |
| P02 | Tight coupling | warning | Два узла с ≥ 3 связями между собой |
| P03 | Missing cache | warning | БД с высоким read-ratio (≥ 0.7) без связанного кэша |
| P04 | Direct UI → State | warning | Страница напрямую связана со store (минуя компоненты) |
| P05 | Stateful service chain | warning | Цепочка из ≥ 2 stateful сервисов |
| P06 | Excessive nesting | warning | UI Component с > 5 вложенными компонентами |
| P07 | No error boundary | info | Frontend без обработки ошибок внешних вызовов |
| P08 | Monolith API | warning | Один API Gateway обслуживает > 15 эндпоинтов |
| P09 | Edge semantics | warning | Некорректная семантика связи (напр. reads/writes для БД, calls для сервисов) |

#### 3.2.4 Правила анализа — Layer 3: Load Estimation

| ID | Правило | Severity | Описание |
|----|---------|----------|----------|
| L01 | Render pressure | warning | Frontend complexity × stateStores > порог → риск тормозов UI |
| L02 | API overload | warning | Суммарный requestRate на gateway > порог |
| L03 | DB write bottleneck | critical | readWriteRatio < 0.3 и > 5 writes-связей |
| L04 | Cache miss impact | warning | Cache hitRate < 0.5 при высокой нагрузке на БД |
| L05 | External dependency risk | warning | External system с reliability < 0.95 на критическом пути |
| L06 | Sync chain latency | warning | Цепочка из ≥ 3 синхронных вызовов (estimated latency) |

#### 3.2.5 Оценка (Score)

Итоговая оценка: **0–100 баллов**.

Формула:
```
base_score = 100
penalty = Σ(issue.severity_weight × issue.count)
  - info:     2 очка
  - warning:  5 очков
  - critical: 15 очков
bonus:
  + cache_present: +3
  + gateway_present: +3
  + no_cycles: +5
  + all_nodes_connected: +3

score = clamp(base_score - penalty + bonus, 0, 100)
```

#### 3.2.6 Результат анализа — структура данных

```typescript
interface AnalysisResult {
  summary: {
    score: number;          // 0–100
    issuesCount: number;
    criticalIssuesCount: number;
    grade: 'A' | 'B' | 'C' | 'D' | 'F';  // score-based
  };
  metrics: ArchitectureMetrics;
  issues: ArchitectureIssue[];
  // Future: aiRecommendations: string[];
  generatedAt: string;
  modelVersion: string;
}
```

### 3.3 Страница анализа (интерактивный отчёт)

#### 3.3.1 Секции страницы

1. **Сводка** — Score (большой, с цветом: зелёный/жёлтый/красный), grade (A–F), количество issues по severity
2. **Метрики** — ключевые числа (узлы, связи, сложность по слоям, нагрузочные оценки)
3. **Список замечаний** — фильтрация по severity и category; сортировка по severity (critical first)
4. **Навигация к проблеме** — при клике на issue → переход в редактор с подсветкой проблемных нод/рёбер

#### 3.3.2 Навигация «Issue → Редактор»

| Шаг | Поведение |
|-----|-----------|
| Пользователь кликает на issue | Приложение сохраняет `affectedNodes` и `affectedEdges` в store |
| Переход в редактор | `navigate('/editor?highlight=nodeId1,nodeId2')` |
| Редактор открывается | Canvas центрируется на проблемных нодах; они визуально подсвечены (пульсация/обводка) |
| Подсветка снимается | По клику на пустое место canvas или через N секунд |

**Acceptance Criteria:**
- Клик по issue в отчёте переносит в редактор за < 300ms
- Проблемные ноды визуально отличимы от обычных (красная/оранжевая обводка)
- Canvas автоматически центрируется и zoom'ится, чтобы все проблемные ноды были видны

### 3.4 Импорт / Экспорт / Сохранение

| Функция | Описание | Формат |
|---------|----------|--------|
| Сохранение | В localStorage браузера | JSON (автоматически) |
| Экспорт | Скачивание файла | `.json` |

**JSON-формат схемы (ArchitectureGraph):**

```typescript
interface ArchitectureGraph {
  meta: {
    name: string;
    version: number;
    createdAt: string;
  };
  nodes: ArchitectureNode[];  // типизированные ноды с параметрами
  edges: GraphEdge[];         // типизированные рёбра
}
```

**Acceptance Criteria:**
- Экспортированный JSON можно импортировать обратно без потери данных (round-trip)
- При импорте невалидного JSON — понятное сообщение об ошибке
- При импорте JSON от другой версии схемы — предупреждение о несовместимости

## 6. Принципы UI-дизайна

### 6.1 Общие принципы

- **Минимализм:** Интерфейс не должен отвлекать от проектирования. Минимум хрома, максимум canvas
- **Progressive disclosure:** Базовые действия доступны сразу; продвинутые параметры — по клику/раскрытию
- **Мгновенная обратная связь:** Каждое действие (добавление ноды, создание связи, сохранение) подтверждается визуально
- **Консистентность:** Единый язык дизайна через Shadcn UI + Tailwind

### 6.2 Layout

### 6.3 UX параметров нод

Ключевая проблема: параметры должны быть понятными для пользователя, но аналитически значимыми.

**Решение — Human-Readable Parameters:**

| Технический параметр | UX-представление | Пример |
|---------------------|------------------|--------|
| `complexity: 1–5` | Ползунок с метками: «Минимальная · Низкая · Средняя · Высокая · Очень высокая» | Tooltip: «Оцените, насколько сложна внутренняя логика этого компонента» |
| `criticality: 0–3` | Сегментированный контрол: «Некритичный · Важный · Критичный · Ключевой» | Визуальный индикатор: ⚪⚡🔥💀 |
| `requestRate` | Dropdown: «Низкая (< 10 rps)», «Средняя (10–100 rps)», «Высокая (100–1000 rps)», «Экстремальная (> 1000 rps)» | Цветовой индикатор нагрузки на ноде |
| `readWriteRatio` | Визуальный бар: «Больше чтения ←→ Больше записи» | Подсказка: «Для аналитических БД — ближе к чтению, для логов — к записи» |
| `hitRate` | Процентный ползунок с зонами: красная (< 50%), жёлтая (50–80%), зелёная (> 80%) | Текст: «Низкий hit rate → кэш неэффективен» |
| `reliability` | Dropdown: «Низкая (< 95%)», «Стандартная (95–99%)», «Высокая (99–99.9%)», «Максимальная (> 99.9%)» | Маппинг на SLA-уровни |

## 9. Потенциальные проблемы и способы их решения

| Проблема | Влияние | Решение |
|----------|---------|---------|
| **UX параметров** — пользователи не понимают, что настраивают | Анализ теряет ценность | Human-readable controls: ползунки с метками, dropdown вместо чисел, tooltips с примерами (см. раздел 6.3) |
| **Формализация паттернов** — сложно перевести архитектурные знания в код-правила | Анализ даёт ложные срабатывания | Начать с простых, детерминированных правил; каждое правило — отдельный модуль с тестами; пороговые значения вынести в конфигурацию |
| **Производительность canvas** — при > 50–100 нодах React Flow может тормозить | Плохой UX для сложных схем | Виртуализация нод (React Flow поддерживает); ленивый рендеринг; ограничение макс. нод с предупреждением |
| **Валидация JSON** — невалидный импорт может сломать приложение | Крэши, потеря данных | Zod-схема с strict mode; fallback на дефолтные значения; показать пользователю конкретное поле с ошибкой |
| **Масштабирование анализа** — при увеличении числа правил время анализа растёт | Медленный отклик на кнопку «Анализ» | Анализ на бэке — не блокирует UI; loading state на фронте; при > 2сек — progress indicator |
| **Качество AI-рекомендаций** (future) | Бесполезные или неверные советы | Гибридный подход: rule-based + AI; AI как дополнение, не замена; возможность скрыть AI-блок |

## 8. Дополнения и требования (2026-02-20)

### 8.1 Scope MVP

- MVP без авторизации. Аутентификация — возможное расширение в будущем.
- UI — минималистичный, без перегруженных цветов и декора.

### 8.2 Улучшения редактора

- **Multi-select**: выделение группы нод (Ctrl+клик / рамка) и совместное перемещение.
- **Drag-and-drop из сайдбара**: добавление нод рядом с курсором.
- **Счётчик элементов/связей** в редакторе (ненавязчивый).
- **Контекстное меню для связей** (ПКМ): редактировать/удалить.
- **Редизайн панели свойств связи**: современный и лаконичный вид.
- **Стили нод и связей**:
  - ноды — более прозрачные, без лишних иконок;
  - связи — нейтральные цвета через CSS‑переменные (готовность к темам).
- **Уникальный корневой узел `system`**:
  - один на схеме;
  - отдельные параметры (название системы, окружение);
  - страницы логически привязаны к нему.
- **Кастомные имена нод** с сохранением типа (type‑badge или подсказка).

### 8.3 Улучшения анализа

- **Больше метрик** и объяснения формул (tooltips).
- **Пересмотр score/grade**: строгая и реалистичная оценка для «плохих» схем.
- **Приоритизация issues**: «что исправить первым».
- **rulesVersion** в ответе анализа для совместимости.

### 8.4 Валидации и ограничения

- **Допустимые связи** между типами нод.
- **Блокировка невозможных связей** на клиенте.
- **Валидация импорта**: предупреждения о некорректных ребрах.

### 8.5 Экспорт и шаблоны

- **Экспорт в PNG/SVG** для вставки в документы.
- **Шаблоны архитектур** (CRUD, microservices, event‑driven).
- **Поиск по графу** и быстрый переход к ноде.

## Приложение B: Glossary

| Термин | Определение |
|--------|------------|
| **Нода (Node)** | Элемент архитектуры на canvas (сервис, БД, компонент и т.д.) |
| **Ребро (Edge)** | Связь между двумя нодами (вызов, чтение, подписка и т.д.) |
| **Граф (Graph)** | Совокупность всех нод и рёбер — полная схема архитектуры |
| **Issue** | Проблема, обнаруженная анализатором |
| **Score** | Итоговая оценка качества архитектуры (0–100) |
| **Grade** | Буквенная оценка: A (90–100), B (75–89), C (60–74), D (40–59), F (0–39) |
| **Layer** | Архитектурный слой: Frontend, Backend, Data |
| **Rule** | Одно правило анализа, проверяющее конкретный аспект архитектуры |
| **Fan-in** | Количество входящих связей у узла |
| **Fan-out** | Количество исходящих связей у узла |

  4. УЗКИЕ МЕСТА И НА ЧТО ОБРАТИТЬ ВНИМАНИЕ
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  ⚠  UX ПАРАМЕТРОВ НОД
     Сейчас все параметры — голые number input'ы. Пользователь видит
     "Complexity: [1]" и не понимает, что это значит. Это ГЛАВНАЯ UX-проблема.
     Задачи TASK-009 и TASK-010 — ключевые для юзабилити.
     Файл: client/src/pages/editor/ui/properties/node-properties-sheet.tsx

  ⚠  МОНОЛИТНЫЙ ANALYSIS SERVICE
     Все 4 правила анализа зашиты inline в один метод analyze().
     Добавлять новые правила в таком формате — хрупко и нетестируемо.
     TASK-001 (рефакторинг в rule engine) — критический путь.
     Файл: server/src/analysis/analysis.service.ts

  ⚠  ОТСУТСТВИЕ ВАЛИДАЦИИ НА БЭКЕ
     DTO (architecture-graph.dto.ts) не имеют декораторов class-validator.
     Любой мусор проходит в analyze(). Потенциально может крашить сервер.
     Файл: server/src/analysis/dto/architecture-graph.dto.ts

  ⚠  ТИПЫ НЕ СИНХРОНИЗИРОВАНЫ
     Frontend (shared/model/types.ts) и backend (dto + analysis.service.ts)
     определяют одни и те же интерфейсы независимо. При расхождении — баги.
     TASK-030 — синхронизация.

  ⚠  НЕТ ВИЗУАЛЬНОГО РАЗЛИЧИЯ НОД
     Все ноды на canvas выглядят одинаково (default тип React Flow).
     Пользователь не видит разницы между Database и UI Component.
     TASK-012 — кастомные ноды с цветами и иконками.

  ⚠  EDGE PARAMETERS НЕ ИСПОЛЬЗУЮТСЯ
     Поля frequency и synchronous определены в типе GraphEdge,
     но не учитываются ни в UI edge-properties, ни в анализе.
     Связи TASK-023 → TASK-007 (L06 sync chain latency).

  ⚠  НЕТ ОБРАБОТКИ ОШИБОК ЗАПРОСА АНАЛИЗА
     При недоступном бэкенде или ошибке — ничего не происходит.
     Нет loading state, нет error state. TASK-027.

  ⚠  ПРЯМОЙ ДОСТУП К /analysis
     Если пользователь перейдёт на /analysis без предварительного анализа,
     страница будет пустой. Нужен guard/redirect. TASK-026.

  Репозиторий:    монорепо (client + server в одном git)
  Ветка:          main
  Начальный коммит: 84e0d56 init: project setup with editor, analysis and docs

  Формат коммитов (Conventional Commits):
    feat:     новая функциональность
    fix:      исправление бага
    refactor: рефакторинг без изменения поведения
    style:    стили, форматирование
    docs:     документация
    chore:    вспомогательное (зависимости, конфиги)
    test:     тесты

  Стратегия веток:
    Сейчас:   работа на main (до MVP)
    После MVP: main (стабильный) ← develop ← feat/TASK-XXX-описание

  Правила:
    - Коммиты крупных фич — только после апрува пользователя
    - Перед коммитом — проверка tsc --noEmit и eslint
    - Не коммитить .env файлы (только .env.example)

  Frontend:  React 19 + TypeScript + Vite
             React Flow (@xyflow/react) для canvas
             Zustand для state management
             Shadcn UI + Tailwind CSS для компонентов
             React Router v7 (lazy loading)
             Zod для валидации, TanStack React Query для fetching

  Backend:   NestJS + TypeScript
             Пока один модуль: AnalysisModule (controller + service)
             Нет БД (stateless), нет auth

  Структура client/src:
    app/         — router, providers, main
    pages/       — editor (canvas, header, properties, store), analysis
    features/    — architecture-graph-io (import/export)
    shared/      — ui (shadcn), model (types), api, lib, hooks, config

  Структура server/src:
    analysis/    — controller, service, dto
    app.*        — root module

# Code Style Guide — ArchLens

Правила код-стайла и архитектурные соглашения проекта.
Цель — читаемость, предсказуемость и масштабируемость.

> Дополняется по ходу разработки. Каждое правило появилось по причине.

---

## 1. Принципы проектирования

### DRY — Don't Repeat Yourself

- Если одно и то же используется в 2+ местах — выноси в `shared/`.
- Общие типы, утилиты, конфиги, UI-компоненты — всё переиспользуемое живёт в `shared/`.
- На бэке: общие интерфейсы в `interfaces/`, конфиги в отдельных файлах.

### KISS — Keep It Simple, Stupid

- Читаемость важнее краткости. Код читается чаще, чем пишется.
- Явные имена лучше сокращений. Понятные конструкции лучше «умных».
- Не добавляй абстракцию, пока она не нужна в текущий момент.

### YAGNI — You Aren't Gonna Need It

- Не пиши код «на будущее». Реализуй только то, что нужно сейчас.
- Заготовки и заглушки допустимы только если они зафиксированы в PRD/tasks.
- Удаляй мёртвый код, не комментируй — git помнит.

### SOLID

- **S — Single Responsibility.** Один файл — одна ответственность. Компонент = UI. Store = состояние + actions. Типы = только типы. Утилиты = чистые функции без побочных эффектов.
- **O — Open/Closed.** Модули открыты для расширения, закрыты для модификации. Пример: rule engine — новое правило = новый файл, без изменения engine.
- **L — Liskov Substitution.** Все реализации `AnalysisRule` взаимозаменяемы. Интерфейс = контракт.
- **I — Interface Segregation.** Маленькие, точечные интерфейсы лучше одного большого. `GraphContext` вместо передачи всего графа в каждую функцию.
- **D — Dependency Inversion.** Зависимость от абстракций (интерфейсов), не от конкретных реализаций. Service зависит от `AnalysisRule[]`, не от конкретных классов правил.

### Дополнительно

- **Composition over Inheritance.** Компоненты собираются из частей, не наследуются.
- **Fail Fast.** Валидируй на входе (Zod на фронте, class-validator на бэке), не прокидывай невалидные данные вглубь.
- **Строгая типизация.** `any` запрещён. `unknown` — только с последующим сужением. Все публичные API типизированы.

## 2. Структура проекта (FSD)

```
src/
├── app/          — инициализация, провайдеры, роутер
├── pages/        — страницы (роуты), page-level логика
├── features/     — изолированные фичи и сценарии
└── shared/       — переиспользуемые ресурсы
    ├── ui/       — UI-компоненты (Shadcn, кастомные)
    ├── model/    — общие типы, конфиги
    ├── api/      — API-клиент, запросы
    ├── lib/      — утилиты
    ├── hooks/    — общие хуки
    └── config/   — env, константы
```

Внутри модуля (page / feature):

```
feature-name/
├── model/        — store, types, selectors
├── ui/           — компоненты
├── lib/          — хелперы, хуки модуля
└── index.ts      — публичный API модуля
```

## 3. Именование

### Файлы и папки

`kebab-case` для всех файлов и директорий:
`node-properties-sheet.tsx`, `use-canvas-handlers.ts`, `architecture-graph.dto.ts`

### Переменные и функции

`camelCase`: `buildExportableGraph`, `handleAnalysisClick`

### Типы и интерфейсы

`PascalCase`: `ArchitectureNode`, `AnalysisResult`, `EdgeKind`

### Константы

`UPPER_SNAKE_CASE` для немутабельных констант: `EDGE_KIND_LABELS`, `NODE_LABELS`
`camelCase` для мутабельных: `const maxRetries = 3`

## 4. TypeScript

### Interface vs Type

- `interface` — для объектов и расширения (extends)
- `type` — для union, primitives, mapped types, утилитарных типов

### Utility types

Использовать `Partial`, `Pick`, `Omit`, `Record` вместо дублирования типов.
Явно указывать назначение patch-объектов:

```typescript
type NodePatch = Partial<Omit<ArchitectureNode, 'id' | 'kind' | 'layer'>>;
```

### Запреты

- `enum` запрещён — использовать union types: `type EdgeKind = 'calls' | 'reads' | ...`
- `any` запрещён
- `as` — только при доказуемой безопасности (type assertion), предпочитать type guards

## 5. React-компоненты

### Структура

- Функциональные компоненты 
- Один компонент — один файл (допускаются мелкие вспомогательные в том же файле)
- Минимум логики в UI — выносить в хуки и утилиты

```typescript
type NodeCardProps = {
  node: ArchitectureNode;
  onSelect: (id: string) => void;
};

export const NodeCard = ({ node, onSelect }: NodeCardProps) => {
  // ...
};
```

### Props

- Явный type для props (не inline)
- Деструктуризация в параметрах
- Дефолтные значения через деструктуризацию, не `defaultProps`

### Хуки

- Не бойся выносить логику в кастомные хуки
- `useMemo` и `useCallback` — только когда действительно необходимо (референсная стабильность для дочерних компонентов, тяжёлые вычисления). Аргументировать при использовании.
- `useEffect` — минимизировать; предпочитать event handlers и derived state

### Экспорт

- Named exports всегда (не default)
- `index.ts` как публичный API модуля — экспортировать только то, что нужно снаружи

## 6. State Management (Zustand)

- Один store на логический домен (editor store, analysis store)
- Actions внутри store, не снаружи
- Selectors выносить в отдельный файл (`selectors.ts`)
- Не подписываться на весь store — использовать точечные селекторы

```typescript
const nodes = useArchitectureStore((s) => s.nodes);
```

## 7. Backend (NestJS)

- Один module — один домен (AnalysisModule)
- Service содержит бизнес-логику, Controller — только маршрутизация
- DTO — отдельные классы с валидацией
- Конфигурация — в отдельных файлах, не захардкожена в сервисах

## 8. Стилизация

- Tailwind CSS — utility-first
- `cn()` (clsx + tailwind-merge) для условных классов
- Shadcn UI компоненты как база
- Поддержка тёмной и светлой темы (CSS-переменные через Tailwind)
- Цвета слоёв: Frontend `blue`, Backend `green`, Data `amber` — используем мягкие оттенки с прозрачностью (`bg-blue-50/60`)

## 9. Принципы дизайна UI

### Минимализм

- Интерфейс должен быть чистым и не перегруженным. Каждый элемент на экране должен нести смысл.
- Не добавлять декоративные элементы (иконки, бордеры, тени) без конкретной цели.
- Визуальная иерархия через размер и вес шрифта, а не через обилие цветов и рамок.

### Интуитивность

- Пользователь не должен думать, что означает поле. Лейбл + tooltip = обязательный минимум.
- Контролы подбираются по смыслу: дискретные значения → ползунок/сегменты, категории → dropdown, да/нет → toggle.
- Абстрактные числовые параметры заменяются на человеко-понятные шкалы с подписями.

### Цвета

- Минимум цветовой палитры. Основа — нейтральные тона, цвет используется для семантики:
  - Слои (frontend/backend/data) — мягкие пастельные оттенки с прозрачностью.
  - Связи — единый нейтральный цвет (`slate-400`), различаются только стилем: сплошная, пунктир, штрих, анимация. Никаких ярких акцентов на рёбрах.
  - Severity: `info` — серый/синий, `warning` — жёлтый, `critical` — красный.
- Не использовать слишком яркие или насыщенные цвета. Subtle > Bold.

### Прозрачность и воздух

- Узлы на canvas — полупрозрачный фон, тонкая обводка. Не должны «кричать».
- Достаточные отступы (padding, gap). Не сжимать элементы.
- Sheet/Panel — визуально отделён, но не перетягивает внимание от canvas.

## 10. Практики, усвоенные в процессе

> Секция дополняется по ходу разработки. Каждый пункт — результат реального кейса.

- **Понятный нейминг переменных.** Не используй однобуквенные переменные (`v`, `e`). В callbacks — имя отражает суть: `(event)` для DOM-событий, `(requestRate)` для значения поля. Shorthand property: `{ requestRate }` вместо `{ requestRate: v }`.
- **Линтеры после каждого изменения.** Прогоняй `tsc --noEmit` + `eslint --fix` + `prettier --write` перед тем как считать задачу завершённой.
